Conflict Details (11 found)

CRITICAL Payment Gateway Conflict Fixable
Both WooCommerce Stripe Gateway and WooPayments register payment handlers on the woocommerce_payment_gateways hook at priority 10. During checkout, Stripe's intent creation can be overridden by WooPayments' handler, causing intermittent "Payment processing failed" errors — especially under concurrent requests during sales events.
woocommerce-gateway-stripe vs woocommerce-payments
Recommendation: Adjust the hook priority of WooPayments' gateway registration to priority 20, or deactivate one of the two gateways if you're only using one payment processor. If both are needed for different currencies, a priority-separation patch resolves this cleanly.
CRITICAL Checkout Field Conflict Fixable
CartFlows overrides WooCommerce checkout fields via woocommerce_checkout_fields at priority 10. Elementor Pro's checkout widget hooks the same filter at the same priority, causing CartFlows' custom fields to be silently dropped on pages built with Elementor. Customers on those pages see a broken checkout form.
cartflows vs elementor-pro
Recommendation: Move CartFlows' checkout field filter to priority 20 so it runs after Elementor Pro's modifications. This preserves both plugins' functionality. A one-line mu-plugin patch resolves this permanently.
HIGH Script Collision Fixable
Both Elementor Pro and CartFlows enqueue a script with the handle flatpickr. Elementor loads v4.6.13 while CartFlows loads v4.6.9. On pages where both load, the older version sometimes wins, causing Elementor's date picker to malfunction.
elementor-pro vs cartflows
Recommendation: Dequeue CartFlows' older flatpickr and let Elementor's newer version serve both plugins. Verified compatible.
HIGH Dependency Version Conflict
WooCommerce Stripe Gateway bundles stripe/stripe-php v13.2.0 while MailPoet bundles v10.21.0. Both are loaded without namespace prefixing. Depending on plugin load order, API calls may use the wrong SDK version, causing intermittent Stripe API errors ("Unexpected API version").
woocommerce-gateway-stripe vs mailpoet
Recommendation: Contact MailPoet support to request they namespace-prefix their Stripe SDK (this is a known issue). In the meantime, ensure WooCommerce Stripe Gateway loads before MailPoet by adjusting plugin load order.
HIGH Performance Degradation
When WP Rocket and Custom Order Status Manager are both active, the WooCommerce orders page takes 4.7 seconds to load (baseline: 1.2 seconds). Custom Order Status Manager runs unindexed queries on wp_posts to count orders per status, and WP Rocket's database optimization cron compounds the issue by locking the table.
wp-rocket vs custom-order-status-manager
Recommendation: Add an index on post_status in the orders table and schedule WP Rocket's database optimization outside business hours. With HPOS enabled, this issue resolves entirely.
